数据与模型方案（修订版）

一、识别任务与业务语义
- 本系统围绕“按计划服药识别”构建了两类感知任务：药品目标检测与服药动作判定。
- 药品目标检测负责识别画面中的药品相关对象，当前统一划分为四类：`PILL`（药片/胶囊）、`BLISTER`（泡罩板）、`BOTTLE`（药瓶）、`BOX`（药盒）。
- 服药动作判定负责识别“手部将药品送至口部附近”的行为，用于与目标检测结果进行融合，形成服药事件状态。
- 系统最终输出的业务状态包括 `suspected`、`confirmed`、`abnormal` 三类。其中：
  - `suspected` 表示仅检测到目标或仅检测到动作，或两者证据尚不足以形成高置信结果。
  - `confirmed` 表示检测到药品目标且检测到服药动作，形成较高置信的服药判断。
  - `abnormal` 表示在当前检测过程中未形成有效服药证据，或检测链路出现异常。

二、实际算法架构
- 当前项目的实际实现不是浏览器端直接推理，而是“三服务协同”模式：
  - Vue3 前端负责摄像头采集、录制视频、展示检测结果。
  - Flask 算法服务负责单帧检测和视频检测。
  - Spring Boot 后端负责计划、事件、告警、日志图片和统计。
- 单帧检测流程：
  - 浏览器抓取图像帧并转换为 Base64。
  - 前端调用 Flask `/v1/detections/predict`。
  - Flask 通过 OpenCV 解码图像，调用 ONNX Runtime 执行目标检测，同时执行手部动作判定。
  - Flask 输出统一协议：`status / confidence / actionDetected / targets / traceId / latencyMs`。
- 视频检测流程：
  - 浏览器录制短视频。
  - 前端调用 Flask `/v1/detections/video/predict`。
  - Flask 抽帧后进行逐帧目标检测和动作时间线分析，最终聚合为一次视频检测结果。

三、目标检测模型设计
- 模型选择：
  - 当前项目采用 YOLO 系列轻量目标检测思路，训练脚本与说明位于 `flask-project/train-yolov8/`。
  - 部署阶段使用导出的 ONNX 模型文件 `models/medication-intake.onnx`，在 Flask 中通过 `onnxruntime` 执行推理。
- 选择理由：
  - YOLO 系列在小目标检测和实时性方面具有较好平衡，适合家庭桌面环境中的药品检测任务。
  - ONNX 便于模型跨环境部署，能够较稳定地接入 Python 服务端推理链路。
  - 轻量模型更适合毕业设计原型系统，对硬件要求较低，便于答辩演示。
- 检测类别：
  - `PILL`：药片或胶囊。
  - `BLISTER`：泡罩板。
  - `BOTTLE`：药瓶。
  - `BOX`：药盒。
- 关键参数：
  - 默认输入尺寸为 `640 x 640`。
  - 目标置信度阈值当前按代码配置执行，可在部署中根据误报率继续调优。
  - 非极大值抑制用于去除重复框，提高最终检测框稳定性。

四、动作判定方法设计
- 本系统当前未采用独立的大型动作识别网络，而是使用“手部关键点 + 规则融合”的轻量方案。
- 具体实现思路如下：
  - 优先尝试通过 MediaPipe Hands 获取手部关键点。
  - 结合人脸或口部近似区域，计算手部与口部之间的空间接近关系。
  - 如果连续一段时间满足“手部靠近口部”的规则，则判定为服药动作证据成立。
- 选择理由：
  - 毕设系统的核心目标是构建可演示、可验证的辅助服药闭环，而非做复杂动作识别研究。
  - 基于关键点和规则的方案实现成本低，解释性强，适合论文中说明判定逻辑。
  - 当 MediaPipe 或复杂动作模块不可用时，系统仍可走降级路径，保证演示链路可运行。

五、事件判定规则
- 系统采用目标检测与动作判定融合的方式生成服药事件。
- 融合逻辑可概括为：
  - 检测到目标且检测到动作：输出 `confirmed`。
  - 仅检测到目标或仅检测到动作：输出 `suspected`。
  - 未检测到有效目标与动作组合：输出 `abnormal`。
- 在业务层中，`suspected` 结果通常需要用户进一步确认；`confirmed` 结果可直接形成已服药记录或进入确认流程；`abnormal` 结果会被记录为异常事件，并可进一步触发告警。
- 这种设计兼顾了实时识别需求与业务可解释性，能够将算法结果映射为老年人用药管理场景中的结构化事件。

六、数据集采集与标注建议
- 当前仓库已提供 YOLOv8 训练脚本、数据目录结构和数据准备脚本，但最终论文中仍需补充你自己的实验数据来源。
- 建议论文从以下几个方面描述数据集：
  - 采集环境：家庭桌面、室内自然光或灯光、普通 PC 摄像头。
  - 类别分布：四类药品对象分别采集样本，覆盖不同角度、遮挡、背景和光照条件。
  - 行为样本：采集服药动作短视频，覆盖不同手势、不同持药方式和不同拍摄距离。
  - 标注方式：目标检测使用矩形框标注；动作判定可通过关键帧区间或动作片段进行标注。
- 建议在论文中补充表格：
  - 训练集样本数、验证集样本数、测试集样本数。
  - 各类别样本数分布。
  - 采集设备、分辨率、帧率和标注工具。

七、模型训练与导出说明
- 训练脚本位于 `flask-project/train-yolov8/scripts/train.py`，支持以下核心参数：
  - `--model`：模型大小，默认 `n`。
  - `--epochs`：训练轮数，默认 `100`。
  - `--batch`：批大小，默认 `16`。
  - `--imgsz`：图像尺寸，默认 `640`。
  - `--device`：训练设备，可为 `cpu`、`cuda` 或 `mps`。
- 训练完成后可导出为 ONNX 模型，并复制到项目部署目录中，供 Flask 服务直接加载。
- 在论文中建议补充：
  - 训练环境：CPU/GPU 型号、内存、操作系统、Python 版本。
  - 训练参数：学习率、训练轮数、批大小、图像增强策略。
  - 训练结果：`mAP@0.5`、Precision、Recall、Loss 曲线。

八、统一检测结果协议
- 为了降低前后端耦合，算法服务对单帧检测和视频检测都采用统一的结果语义：
  - `status`：检测结果状态。
  - `confidence`：综合置信度。
  - `actionDetected`：是否检测到动作证据。
  - `targets`：检测到的目标数组，包含类别、置信度和框信息。
  - `traceId`：链路跟踪 ID。
  - `latencyMs`：检测耗时。
- 视频检测还额外输出：
  - `actionTimeline`：动作发生的帧区间。
  - `videoMetadata`：视频时长、帧率、总帧数和处理帧数。
